# Git Reset vs Git Revert

## 1. Ключевая разница в одном предложении

| Команда | Суть |
|---------|------|
| `git reset` | **Перемещает указатель ветки назад**, стирая коммиты из истории |
| `git revert` | **Создаёт новый коммит**, который отменяет изменения выбранного коммита |

```
git reset  = «Машина времени» — переписывает историю
git revert = «Ctrl+Z» — добавляет новую запись в историю, отменяющую старую
```

---

## 2. Как работает `git reset`

### Механизм

`git reset` перемещает указатель текущей ветки (файл в `.git/refs/heads/`) на указанный коммит. Коммиты, оставшиеся «позади», перестают быть достижимыми из ветки.

```
До reset:
    A --- B --- C --- D    ← main (HEAD)

git reset --hard B:
    A --- B    ← main (HEAD)
         (C и D больше не в ветке, но пока живут в .git/objects)
```

### Три режима reset

| Режим | HEAD | Index (Staging) | Working Tree | Применение |
|-------|------|-----------------|--------------|------------|
| `--soft` | ✅ Перемещает | ❌ Не трогает | ❌ Не трогает | Переделать коммит (squash) |
| `--mixed` (по умолч.) | ✅ Перемещает | ✅ Сбрасывает | ❌ Не трогает | Убрать из staging |
| `--hard` | ✅ Перемещает | ✅ Сбрасывает | ✅ Сбрасывает | Полный откат |

### Пример: --soft

```bash
# Хотим объединить 3 последних коммита в один
git reset --soft HEAD~3
git commit -m "Все три изменения в одном коммите"
```

Содержимое Index и Working Tree остаётся как было → можно сразу закоммитить.

### Пример: --hard

```bash
# Полностью откатить последние 2 коммита
git reset --hard HEAD~2
```

⚠️ Изменения в Working Tree **будут потеряны** (но объекты останутся в reflog ~30 дней).

---

## 3. Как работает `git revert`

### Механизм

`git revert` создаёт **новый коммит**, содержащий изменения, обратные указанному коммиту. История не переписывается — она дополняется.

```
До revert:
    A --- B --- C --- D    ← main (HEAD)

git revert C:
    A --- B --- C --- D --- C'   ← main (HEAD)
                            │
                    C' отменяет изменения C,
                    но B и D остаются нетронутыми
```

### Пример

```bash
# Отменить конкретный коммит
git revert abc1234

# Отменить диапазон коммитов (от старого к новому, не включая left boundary)
git revert HEAD~3..HEAD

# Отменить без автоматического коммита (для ручной правки)
git revert --no-commit abc1234
```

## 5. ⚡ Главное отличие: накатывание новых изменений (push/pull)

Это **самая важная практическая разница**. Рассмотрим ситуацию, когда вы работаете с удалённым репозиторием и другими разработчиками.

### Сценарий: коммит D нужно отменить

```
Локально и на origin:
    A --- B --- C --- D    ← main, origin/main
```

### Вариант 1: `git reset` — проблемы при push

```bash
git reset --hard HEAD~1     # Локально: A-B-C
git push origin main        # ❌ ОТКЛОНЕНО! origin впереди на 1 коммит
git push --force origin main # ⚠️ Работает, но...
```

**Что происходит у коллег:**

```
У коллеги Маши (до вашего force push):
    A --- B --- C --- D    ← main
                      ↑
               origin/main

После вашего force push, Маша делает git pull:
    A --- B --- C --- D     ← main (локальная)
                 \
                  ← origin/main (вы откатили до C)

    → КОНФЛИКТ ИСТОРИИ! Git не может сделать fast-forward.
    → Маша получает ошибку или merge-конфликт.
    → Если Маша сделала свои коммиты поверх D — они «повиснут в воздухе».
```

**Ещё хуже:** если Маша сделает `git pull` без `--rebase`, Git создаст merge-коммит, который **вернёт коммит D обратно** в историю:

```
После git pull у Маши (без rebase):
    A --- B --- C --------- M    ← main
                 \         /
                  D ------
                  (коммит D вернулся через merge!)
```

### Вариант 2: `git revert` — чистое накатывание

```bash
git revert HEAD             # Создаёт коммит D', отменяющий D
git push origin main        # ✅ Обычный push, никаких --force
```

**Что происходит у коллег:**

```
У коллеги Маши после git pull:
    A --- B --- C --- D --- D'    ← main, origin/main
                             │
                     D' отменяет D — всё чисто!

    → Обычный fast-forward.
    → Никаких конфликтов.
    → Маша даже не заметит проблем.
```

### Таблица: push/pull поведение

| Аспект | `git reset` + push | `git revert` + push |
|--------|-------------------|---------------------|
| Нужен `--force`? | ✅ Да | ❌ Нет |
| Ломает `git pull` у коллег? | ✅ Да | ❌ Нет |
| Коллеги теряют свои коммиты? | ⚠️ Возможно | ❌ Нет |
| Может вернуть удалённый коммит через merge? | ✅ Да | ❌ Нет |
| Совместим с CI/CD? | ⚠️ Проблемы с protected branches | ✅ Полностью |

## 6. Подводный камень revert: повторное накатывание отменённых изменений

У `git revert` есть один важный нюанс. Если вы отменили коммит через revert, а потом хотите **снова применить** те же изменения — простой `git merge` не поможет:

```
Ситуация:
    A --- B --- C --- R(revert C)    ← main
          \
           C --- D --- E             ← feature

git merge feature в main:
    Git считает, что C уже был в main и был СОЗНАТЕЛЬНО отменён.
    Изменения из C НЕ БУДУТ применены повторно!
    Применятся только D и E.
```

### Решение: revert the revert

```bash
# Сначала отменить revert-коммит
git revert R    # Это вернёт изменения C обратно

# Теперь можно мержить feature
git merge feature   # D и E применятся нормально
```

```
    A --- B --- C --- R --- R' --- M    ← main
          \                      /
           C --- D --- E -------        ← feature

R  = revert C (отменяет C)
R' = revert R (возвращает C обратно)
M  = merge feature (применяет D, E)
```

## 7. Откатывание Merge-коммита через `git revert`

Обычный коммит имеет **одного родителя**. Merge-коммит имеет **двух (или более) родителей**. Это создаёт проблему: при revert Git не знает, относительно какого родителя вычислять обратный diff.

### Проблема: revert без `-m` не работает

```bash
git revert M
# error: commit M is a merge but no -m option was given.
# fatal: revert failed
```

Git требует указать, **какой родитель считать «основной линией» (mainline)**, чтобы понять, какие изменения отменять.

### Что такое `-m` (mainline parent)

У merge-коммита родители пронумерованы:

```
         parent 1         parent 2
            │                 │
            ▼                 ▼
    A --- B --- C --- M    ← main (HEAD)
                     /
          D --- E ---      ← feature

Коммит M имеет:
  parent 1 = C  (ветка, В КОТОРУЮ мержили — main)
  parent 2 = E  (ветка, КОТОРУЮ мержили — feature)
```

Посмотреть родителей можно так:

```bash
git cat-file -p <sha-of-M>
# tree ...
# parent <sha-of-C>    ← parent 1
# parent <sha-of-E>    ← parent 2
# author ...
# committer ...
# Merge branch 'feature' into main
```

### Флаг `-m 1` — отменить изменения из feature

```bash
git revert -m 1 M
```

Это значит: «Считай parent 1 (коммит C в main) основной линией. Отмени всё, что пришло из другой стороны (feature)».

```
До:
    A --- B --- C --- M        ← main
                     /
          D --- E ---          ← feature

git revert -m 1 M:
    A --- B --- C --- M --- M'    ← main
                     /
          D --- E ---             ← feature

M' отменяет все изменения, пришедшие из feature (коммиты D и E).
Состояние кода в M' совпадает с состоянием в C.
```

### Флаг `-m 2` — отменить изменения из main

```bash
git revert -m 2 M
```

Это значит: «Считай parent 2 (коммит E в feature) основной линией. Отмени всё, что пришло из main».

```
git revert -m 2 M:
    Состояние кода в M' совпадает с состоянием в E.
    Отменяются изменения, которые были в main (коммит C), но не в feature.
```

⚠️ В подавляющем большинстве случаев нужен именно **`-m 1`** — вы отменяете влитую feature-ветку.

### Полный пример: сломанный merge в production

```bash
# Ситуация: feature влили в main, и она сломала production
git log --oneline --graph
# *   abc1234 (HEAD -> main) Merge branch 'feature' into main
# |\
# | * def5678 (feature) Add new payment system
# | * 111aaaa Refactor checkout
# |/
# * 222bbbb Last stable commit

# Откатываем merge, сохраняя main как mainline
git revert -m 1 abc1234
# [main 333cccc] Revert "Merge branch 'feature' into main"

# Пушим — никакого --force!
git push origin main
```

### ⚠️ Подводный камень: повторный merge после revert merge

После того как вы откатили merge через `revert -m 1`, Git считает, что коммиты из feature **уже были интегрированы**. Если вы попробуете снова влить ту же feature-ветку — **изменения не применятся**:

```bash
# feature починили и хотим влить снова
git merge feature
# Already up to date.  ← ИЛИ применятся только НОВЫЕ коммиты после E
```

```
    A --- B --- C --- M --- M' --- ???
                     /
          D --- E --- F(fix)

git merge feature:
    Только F будет влит!
    D и E Git считает уже обработанными (и отменёнными).
```

### Решение: revert the revert, затем merge

```bash
# 1. Отменяем revert-коммит (возвращаем изменения feature)
git revert M'    # M'' — отмена отмены, изменения D+E возвращаются

# 2. Теперь мержим feature с исправлениями
git merge feature    # F применяется нормально

# Итоговая история:
#   A --- B --- C --- M --- M' --- M'' --- N    ← main
#                    /                    /
#         D --- E -------- F ------------       ← feature
#
# M   = merge feature (D+E вошли)
# M'  = revert merge  (D+E отменены)
# M'' = revert revert (D+E возвращены)
# N   = merge feature (F входит)
```

### Альтернатива: rebase feature перед повторным merge

Если не хотите «revert the revert», можно пересоздать коммиты feature с новыми SHA:

```bash
# Создаём новую ветку из feature и перебазируем
git checkout feature
git rebase --no-ff main
# Теперь D и E получили новые SHA → Git считает их новыми коммитами

git checkout main
git merge feature    # Всё применяется нормально
```

### Сводная таблица: revert merge

| Действие | Команда | Результат |
|----------|---------|-----------|
| Откатить влитую ветку | `git revert -m 1 <merge>` | Код как до merge, история сохранена |
| Откатить основную линию | `git revert -m 2 <merge>` | Код как в feature-ветке (редко нужно) |
| Повторно влить feature | `git revert <revert-commit>` + `git merge feature` | Полное восстановление |
| Посмотреть родителей merge | `git cat-file -p <merge>` | Показывает parent 1, parent 2 |
| Проверить номер parent | `git log --oneline --first-parent` | first-parent = parent 1 |

## 8. Внутреннее устройство: что происходит в `.git`

### При `git reset --hard HEAD~1`:

```bash
# 1. Обновляется файл ссылки ветки
echo "<sha1 предыдущего коммита>" > .git/refs/heads/main

# 2. Обновляется Index (staging area)
#    Index перезаписывается tree-объектом из целевого коммита

# 3. Обновляется Working Tree (только при --hard)
#    Файлы в рабочей директории перезаписываются из Index
```

### При `git revert HEAD`:

```bash
# 1. Вычисляется обратный diff коммита HEAD
#    (diff между tree коммита и tree его родителя — инвертированный)

# 2. Обратный diff применяется к текущему состоянию
#    → создаются новые blob-объекты для изменённых файлов

# 3. Создаётся новый tree-объект

# 4. Создаётся новый commit-объект:
#    - tree: <новый tree>
#    - parent: <текущий HEAD>
#    - message: "Revert \"<сообщение отменяемого коммита>\""

# 5. Указатель ветки перемещается ВПЕРЁД на новый коммит
```

## 9. Когда что использовать

### Используйте `git reset` когда:

- 🔒 Работаете **только локально** (коммиты ещё не были запушены)
- 🧹 Хотите «схлопнуть» несколько коммитов в один (`--soft`)
- 🚫 Хотите убрать файл из staging area (`git reset HEAD file.txt`)
- 🧪 Экспериментируете в одиночку на feature-ветке

### Используйте `git revert` когда:

- 🌍 Коммит **уже опубликован** (запушен в shared-репозиторий)
- 👥 Работаете **в команде** — нельзя ломать чужую историю
- 📋 Нужен **аудит-трейл** — видно, что было и что отменили
- 🛡️ Ветка **защищена** (protected branch, нет force push)

## 10. Спасательный круг: `git reflog`

Если вы случайно сделали `git reset --hard` и потеряли коммиты:

```bash
# Посмотреть историю перемещений HEAD
git reflog

# Пример вывода:
# abc1234 HEAD@{0}: reset: moving to HEAD~3
# def5678 HEAD@{1}: commit: Важный коммит, который я случайно удалил
# ...

# Восстановить потерянный коммит
git reset --hard def5678

# Или создать новую ветку из потерянного коммита
git branch recovery def5678
```

> ⏰ Записи reflog хранятся по умолчанию **90 дней**, после чего `git gc` может удалить осиротевшие объекты.
